# STARCARE 清單頁 UI/UX 規格 v1

> 對齊：`docs/business-logic.md`（閉環與角色權限）、`README.md`（專案收尾脈絡）  
> 目的：統一全 App 清單頁呈現與操作層級，降低認知負擔並提升跨模組一致性。

## 1. 頁面骨架（所有清單頁一致）

1. `Header`：模組名稱 + 一句用途 + 主要 CTA（最多 1 個主色按鈕）。
2. `KPI Strip`：3-5 個核心指標（例如總數、待處理、逾期、今日新增）。
3. `Primary Action Zone`：新增／匯入／批量等會改變狀態的主操作區。
4. `List Zone`：搜尋、篩選、排序、表格、分頁。
5. `Secondary Zone`：審計、技術明細、歷史紀錄（預設收合）。

## 2. 視覺層級規則

- 主標題 > KPI > 主操作 > 主清單 > 次要資訊。
- 同層資訊使用一致字級／間距，避免每區都像主重點。
- 每頁主色按鈕最多 1 個；其餘為次級按鈕。

## 3. 清單表格規則

- 表頭固定（sticky）且欄位語意一致。
- 第一欄為核心識別（姓名／ID），最後一欄為操作。
- 常用操作最多 2 個，其餘放入「更多」選單。
- 長文字截斷並提供 tooltip。
- 空狀態需提供下一步 CTA（不可只顯示「無資料」）。

## 4. 篩選與搜尋規則

- 預設顯示常用條件：關鍵字 + 類型 + 狀態 + 日期。
- 進階條件放入「更多篩選」區。
- 必備「清除篩選」。
- 篩選變更時，列表與 KPI 同步刷新。

## 5. 次要資訊區（收合）規則

- 審計／技術明細／歷史紀錄預設收合。
- 收合狀態仍顯示摘要（例：`顯示 x / y`、`最近更新時間`）。
- 展開後才顯示篩選與完整明細。
- TeamLead/Admin 可記住上次展開狀態；Staff 預設不記憶（避免干擾）。

## 6. 批量操作 UX 規則

- 只有勾選項目時才顯示批量工具列。
- 危險操作（刪除、批量覆蓋）需雙重確認。
- 操作結果要可讀：成功筆數、失敗筆數、可重試方式。

## 7. 角色化顯示規則

- Staff：主流程優先，次要資訊弱化。
- TeamLead：開放進階篩選與批量工具。
- Admin：完整審計與管理功能。
- 不做完全不同頁面，而是控制資訊密度與操作層級。

## 8. 可用性細節

- Loading 使用 skeleton，避免版面大幅跳動。
- 返回清單保留捲動位置與篩選條件。
- 支援鍵盤操作（Tab / Enter / Esc）。
- 窄螢幕切換為卡片式顯示（重要欄位前置）。

## 9. 建議導入順序

### 第一批（高頻核心）
- `#staff-import`
- `#residents`
- `#scheduling`

### 第二批（中高頻）
- `#service-forms`
- `#historical-documents`
- `#notification-center`

### 第三批（低頻或管理向）
- 其餘清單型頁面

## 10. 驗收標準（Definition of Done）

- 使用者可在 3 秒內定位主操作區。
- 次要資訊預設不干擾主流程。
- 同角色跨頁操作模型一致。
- 主要任務步驟數不增加（或下降）。
- 首次使用者可在少量說明下完成核心任務。
